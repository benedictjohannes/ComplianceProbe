# Remote Playbook & Secure Report Submission 🌐

ComplianceProbe supports a "Connected" mode where playbooks are fetched from a remote source and results are securely submitted back to a central server. This enables fleet-wide compliance monitoring and centralized reporting.

## 📥 Loading Remote Playbooks

You can load a playbook directly from a secure URL instead of a local file. This allows you to maintain a "Single Source of Truth" for your compliance standards.

### Usage
Pass the HTTPS URL as the first argument to the probe:

```bash
./compliance-probe https://config.internal.company.com/playbooks/security-v2.yaml
```

### Notes
- **HTTPS Only**: For security reasons, loading playbooks over insecure `http://` is strictly prohibited and will result in an error.
- **Timeout**: The probe has a default timeout of 60 seconds for fetching remote configurations.

## 📤 Remote Report Submission

By default, ComplianceProbe saves reports to the local `reports/` directory. However, you can configure it to submit reports to a remote HTTP endpoint.

### Configuration
In your `playbook.yaml`, set the `reportDestination` and provide the endpoint details:

```yaml
# Set destination to 'https'
reportDestination: https

# Configuration for remote submission
reportDestinationHttps:
  url: https://api.compliance-hub.com/v1/submit
  # Format can be 'multipart' (default) or 'json'
  format: json
  # Optional: Secret used to sign the payload (HMAC-SHA256)
  signatureSecret: "your-extremely-long-secure-secret"
  # Optional: Custom headers for authentication or tracking
  additionalHeaders:
    X-Device-ID: "FIN-AUDIT-001"
    Authorization: "Bearer your-api-token"
```

### Submission Formats

ComplianceProbe supports two formats for remote submission: `multipart` (legacy/default) and `json`.

#### 1. Multipart Format (`format: multipart`)
Reports are submitted via an **HTTPS POST** request using `multipart/form-data`. This is the default format.

| Part Name     | Content Type       | Description                                          |
| :------------ | :----------------- | :--------------------------------------------------- |
| `report.json` | `application/json` | Machine-readable structured results (Base64 encoded) |
| `report.md`   | `text/markdown`    | Human-readable summary (Base64 encoded)              |
| `report.log`  | `text/plain`       | Full execution trace (Base64 encoded)                |

> [!NOTE]
> All file contents are **Base64 encoded** within the multipart form parts for consistency across formats.

#### 2. JSON Format (`format: json`)
Reports are submitted as a single **HTTPS POST** request with `application/json` content type. The structure of this "envelope" is defined as the `RemoteSubmission` interface in [`submission.d.ts`](./submission.d.ts).

**Outer Submission Payload ("The Envelope"):**
See [`RemoteSubmission`](./submission.d.ts) for the full payload structure. It contains Base64 encoded versions of `json`, `md`, and `log`, along with their optional HMAC-SHA256 signatures.

**Inner Report JSON Schema:**
The `json` field in the payload above, once Base64-decoded, contains a structured JSON report. The schema for this inner report (including execution times, assertion results, and stats) is defined as the `FinalReport` interface in [`submission.d.ts`](./submission.d.ts).

### 🔐 Security & Payload Signing
If `signatureSecret` is configured, ComplianceProbe will calculate an **HMAC-SHA256** signature for each file. This allows the receiving server to verify that the report was generated by an authorized agent and has not been tampered with.

The signature is calculated against the **Base64 encoded content** of the respective file (the "inner" data) and sent as a hex-encoded string.

#### Signature Delivery
- **Multipart**: Signatures are sent as additional form parts (e.g., `report.json.signature.txt`).
- **JSON**: Signatures are sent as sibling fields (e.g., `jsonSignature`).

#### Server-side Verification Logic (Go Pseudo-code)
```go
// 1. Get the base64 content from the envelope
// (From multipart part OR JSON field)
content := payload.JSON 

// 2. Get the expected signature from the envelope
// (From '.signature.txt' part OR 'jsonSignature' field)
expectedSig := payload.JSONSignature

// 3. Calculate HMAC-SHA256(content, secret)
// Note: We sign the Base64 string directly
mac := hmac.New(sha256.New, []byte(secret))
mac.Write([]byte(content))
calculatedSig := hex.EncodeToString(mac.Sum(nil))

// 4. Compare (using constant-time comparison)
if hmac.Equal([]byte(calculatedSig), []byte(expectedSig)) {
    // Signature valid! Now you can safely Base64-decode 'content'
    // and parse it using the exported types package `report` or submission.d.ts 
}
```
